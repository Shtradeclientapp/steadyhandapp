## Mount → Render Trace

### Phase 1: SSR (server)
`DemoSwitcher` is a `'use client'` component rendered in `app/layout.tsx` (a Server Component). On the server, React renders it with initial state: `profile = null`. Line 83 fires `return null` → **zero bytes in the HTML**. No space is reserved in the layout.

### Phase 2: Client mount + `useEffect`

```
Mount → render null (profile===null)
     → useEffect fires (after paint)
       → createClient() → new createBrowserClient instance
       → getSession()   [async, network/cookie read]
```

**Timing gap #1 — entire duration is invisible.**  
The component returns `null` for the full duration of initialization. Because `layout.tsx` renders `<DemoSwitcher />` before `{children}`, when it does eventually appear it pushes all page content down 40px, causing a layout shift.

---

### Phase 3a: Session found on first call (happy path)

```
getSession() → session ✓
  → profiles.select() [async DB round-trip]
    → data.is_demo === true
      → setProfile + setRole → re-render → bar visible
```

**Timing gap #2 — two sequential async calls.**  
`getSession()` + `profiles.select()` run serially. Typical latency: 200–600ms. The bar is invisible the whole time.

---

### Phase 3b: Session not available on first call (hydration race)

```
getSession() → null
  → setTimeout(..., 1000)  ← hard 1-second wait
  → return                 ← exits, component stays null
  [1000ms pass]
  → getSession() again
    → null: return (give up, bar never appears) ← silent failure
    → session ✓: profiles.select() → setProfile → bar visible
```

**Timing gap #3 — mandatory 1-second blind wait.**  
`@supabase/ssr`'s `createBrowserClient` reads the session from cookies/localStorage asynchronously. The first `getSession()` call during hydration often returns `null` not because there's no session, but because the client hasn't finished parsing storage yet. The component unconditionally waits a full second before retrying, even if the session becomes available in 50ms.

**Silent failure #1 — if both calls return null, the bar never appears.** No error is thrown, no state is set, no retry after the second failure.

---

### Phase 4: Profile fetch

Both the happy path and the retry path do:
```ts
const { data } = await supabase
  .from('profiles')
  .select('*, tradie:tradie_profiles(business_name)')
  .eq('id', session.user.id)
  .single()
```

**Silent failure #2 — `error` is destructured away and ignored.** If the query fails (RLS denial, network error, no matching row), `data` is `null`, `is_demo` check is skipped, and `setProfile` is never called. The bar stays invisible with no indication of why.

---

## Summary of issues

| # | Issue | Effect |
|---|-------|--------|
| 1 | No loading/placeholder state; `return null` from first render | Layout shift + bar invisible during all init |
| 2 | Two sequential async calls (session → profile) on every mount | 200–600ms blank even for valid demo sessions |
| 3 | Hard 1s `setTimeout` retry instead of `onAuthStateChange` listener | Mandatory 1s delay when session isn't immediately in storage |
| 4 | Only one retry; silent `return` on second failure | Bar permanently hidden if session takes >1s to hydrate |
| 5 | Profile query error silently swallowed | Bar hidden with no observable signal if DB call fails |

---

## The fix

Replace the timeout polling with an `onAuthStateChange` listener, which fires reliably when the session is ready regardless of hydration timing:

```ts
useEffect(() => {
  const supabase = createClient()

  const loadProfile = async (userId: string) => {
    const { data, error } = await supabase
      .from('profiles')
      .select('*, tradie:tradie_profiles(business_name)')
      .eq('id', userId)
      .single()
    if (error || !data?.is_demo) return
    setProfile(data)
    setRole(data.role === 'tradie' ? 'tradie' : data.org_id ? 'org' : 'client')
  }

  // Check current session immediately
  supabase.auth.getSession().then(({ data: { session } }) => {
    if (session) loadProfile(session.user.id)
  })

  // Also listen for auth state changes (fires when session hydrates)
  const { data: { subscription } } = supabase.auth.onAuthStateChange((_event, session) => {
    if (session) loadProfile(session.user.id)
    else setProfile(null)
  })

  return () => subscription.unsubscribe()
}, [])
```

This eliminates the 1-second hardcoded wait and handles all hydration timings — the listener fires as soon as the session is available, whether that's 10ms or 800ms after mount.

## Scan 10: Search for all webhook handlers and check whether the idempotency check on processed_stripe_events is applied to every event type

There's only one webhook file. The idempotency check is applied at the handler level, so coverage across event types is fine — but there's a critical ordering bug.

**Coverage: all event types are protected**

The idempotency guard (lines 25–35) runs before the `switch` statement, so every event type — `payment_intent.succeeded`, `payment_intent.payment_failed`, `transfer.created`, `account.updated`, `checkout.session.completed`, `customer.subscription.deleted`, and even unhandled events hitting `default` — all pass through the same check.

**The bug: event is recorded before processing**

Line 35 inserts into `processed_stripe_events` *before* the `try` block at line 37 executes any business logic. The failure path is:

1. Event arrives, not in table → insert row (line 35)
2. Processing throws (e.g. Supabase error inside the `try`)
3. Handler returns 500
4. Stripe retries the webhook
5. Event **is** in the table now → `skipped: true`, retry silently dropped

This means any transient error during processing permanently blocks that event from being re-processed. The fix is to only record the event after successful processing, or use a status column (`pending` → `completed`) and only skip `completed` events.

Suggested fix for `route.ts`:

```ts
// Move the insert to AFTER the switch succeeds, inside the try block
try {
  switch (event.type) {
    // ... all cases ...
  }

  // Only record after successful processing
  await supabase
    .from('processed_stripe_events')
    .insert({ event_id: eventId, event_type: event.type, created_at: new Date().toISOString() })

  return NextResponse.json({ received: true })
} catch (err: any) {
  // Don't insert — Stripe will retry and we'll try again
  console.error('Webhook handler error:', err.message)
  return NextResponse.json({ error: err.message }, { status: 500 })
}
```
