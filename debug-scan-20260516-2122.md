## Scan 5: Trace the DemoSwitcher component from mount to render and identify any async timing issue that would cause it not to appear

Here's the full trace and the timing bugs:

---

## Mount → Render Trace

**1. Server render**
`RootLayout` is a Server Component. `DemoSwitcher` is `'use client'`, so the server emits only a component shell — no content renders server-side.

**2. Client hydration — initial render**
`profile = null`, so line 83 (`if (!profile) return null`) fires immediately. The bar is invisible.

**3. `useEffect` fires (post-paint)**
`createClient()` is called — this creates a **brand-new** `createBrowserClient` instance, which starts cold with no cached auth state. Then `checkProfile()` runs:

**4. `await supabase.auth.getSession()`**
`createBrowserClient` from `@supabase/ssr` reads its session from cookie storage. **But the internal auth module initializes asynchronously** — the first `getSession()` call triggers that initialization and can return `null` even when a valid session cookie exists, because the client hasn't finished bootstrapping its auth state from storage yet.

**5. Branch on null session**
The component detects `!session` and falls into a `setTimeout(..., 1000)` retry, then returns early from `checkProfile`.

**6. 1-second retry fires**
If Supabase has finished loading the session by then, the retry works and the bar appears ~1s after mount. If not (slow device, slow storage read, or the component unmounts before the timeout fires), **the component never appears**.

---

## The Bugs

**Bug 1 — Primary: one-shot `getSession()` races against auth initialization**
`getSession()` is called before Supabase's browser client has loaded the session from cookie storage. The correct pattern is `onAuthStateChange`, which fires a guaranteed `INITIAL_SESSION` event once the session is resolved — no guessing required.

**Bug 2 — Secondary: the 1s retry is the only fallback**
There is no `onAuthStateChange` listener. If the session resolves after 1 second (rare but possible on slow hardware), or if the timeout fires but the session is *still* not ready, the component silently stays invisible forever.

**Bug 3 — Minor: new client instance on every mount**
`createClient()` at line 11 creates a fresh `createBrowserClient` on every `useEffect` call. Each fresh instance starts with no warm cache, making the race condition in Bug 1 more likely to trigger.

---

## The Fix

Replace the `getSession()` + `setTimeout` pattern with `onAuthStateChange`:

```ts
useEffect(() => {
  const supabase = createClient()

  const loadProfile = async (userId: string) => {
    const { data } = await supabase
      .from('profiles')
      .select('*, tradie:tradie_profiles(business_name)')
      .eq('id', userId)
      .single()
    if (data?.is_demo) {
      setProfile(data)
      setRole(data.role === 'tradie' ? 'tradie' : data.org_id ? 'org' : 'client')
    }
  }

  // Fires INITIAL_SESSION once auth state is resolved from storage — no race
  const { data: { subscription } } = supabase.auth.onAuthStateChange((_event, session) => {
    if (session) loadProfile(session.user.id)
  })

  return () => subscription.unsubscribe()
}, [])
```

`INITIAL_SESSION` fires exactly once after the client resolves its session from cookies — no 1s guess, no silent failure. The existing `setTimeout` retry is a symptom patch for this root cause.
